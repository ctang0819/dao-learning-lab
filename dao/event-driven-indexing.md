# 事件驱动索引与增量就绪

## 模型

当候选集合很大、命中对象很少、查询频繁重复时：

1. 预先维护对象与订阅者之间的反向索引。
2. 在状态变化时，只定位并更新真正受影响的订阅者。
3. 维护可直接消费的增量结果集合（就绪队列）。
4. 用持久状态、生命周期管理和扇出处理的复杂度，换取避免反复全量扫描。

## 五要素

| 要素 | 问题 |
| --- | --- |
| 目标 | 希望避免哪一种重复的全量工作？ |
| 约束 | 候选集与活跃集的规模分别是多少？ |
| 状态/结构 | 关注集合、反向索引、就绪集合如何保存？ |
| 机制 | 什么事件触发定向更新？ |
| 边界 | 真实工作、扇出、内存与生命周期成本是什么？ |

## 迁移检查

只有能解释多个场景、能预测设计取舍的规律，才可称为“道”。对每个新场景，明确源对象、状态变化、反向索引、就绪队列和边界的映射。

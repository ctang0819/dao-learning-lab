# 依赖驱动任务调度器

## 学习假设

当 100,000 个任务中只有少数任务因依赖完成而可运行时，维护依赖反向索引和就绪队列的调度器，能避免在每次调度周期扫描所有任务。

## 模型映射

| epoll | 任务调度器 |
| --- | --- |
| socket | 任务 |
| 可读/可写状态变化 | 依赖完成、配额释放或工具恢复 |
| 关注集合 | 等待条件满足的任务集合 |
| socket 到 epoll 的反向关联 | 前置任务到后继任务的依赖图 |
| 就绪队列 | 可执行任务队列 |
| `epoll_wait` | worker 获取下一批可执行任务 |

## 已实现的最小范围

实现一个可追踪的 Python CLI，支持：

1. 注册任务和依赖边。
2. 维护每个任务的 `remainingDependencies`。
3. 在前置任务完成时，仅遍历其后继任务；计数降至零才将任务入就绪队列。
4. 输出扫描次数、受影响任务数和就绪队列长度。
5. 提供一个朴素基线：每次完成事件后扫描全部任务。

## 运行与测试

```bash
cd projects/dependency-driven-scheduler
python3 demo.py
python3 -m unittest discover -s tests -v
```

`demo.py` 构造 100,000 个候选任务，其中只有 100 个依赖 `root`。它验证两种实现得到相同的就绪任务，并输出增量更新量与全量扫描量。

当前演示结果：一次 `root` 完成事件中，增量实现更新 100 个真实后继任务；全量扫描基线检查 100,000 个候选任务。两者返回相同的就绪任务集合。

## 验收场景

| 场景 | 期望 |
| --- | --- |
| 100,000 个任务，少数依赖完成 | 增量调度器仅处理相关后继任务；基线扫描全部任务。 |
| 一个前置任务有 10,000 个后继 | 增量调度器处理这 10,000 个真实受影响者，并明确记录扇出成本。 |
| 一个任务依赖 A、B、C | 仅 A、B、C 均完成后，该任务才进入就绪队列。 |

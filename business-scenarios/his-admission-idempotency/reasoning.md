# 逐步推理

## 1. 入口防抖不是业务幂等

虚拟入院登记接口 `POST /admissions` 是写操作。用户连点、客户端超时重试、网关重放或两个应用实例并发接收，都可能让同一业务意图多次到达。

`@RepeatedSubmit` 一类入口策略可改善连点体验，但其跨实例能力、识别规则与时间窗口取决于具体实现。因此它不能作为“绝不重复入院”的最终保证。

## 2. 先定义业务不变量

业务需要区分两件不同的事：

```text
同一患者可有多次历史入院
但同一机构内，同一患者只能有一条有效在院记录
```

若仅在 Java 服务中执行“查询在院记录，再插入登记”，两个并发请求都可能在查询时看不到记录，然后各自插入。这是 `check-then-act` 竞态。

最终裁决应落在共享且原子的数据状态上，例如独立的 `admission_active_guard`：

```text
org_id + patient_id 唯一
```

创建登记的本地事务同时占用 guard 并写入入院记录；出院时在同一事务释放 guard。唯一键冲突的一方查询既有登记并按幂等语义返回结果。

## 3. 本地事务不能覆盖医保

本地数据库与医保系统不共享事务。把医保调用简单地放在数据库写入前或后，都保留失败窗口：

```text
医保成功 -> 本地提交失败：外部已登记，本地无记录
本地提交成功 -> 医保失败：本地已登记，外部未完成
```

因此需要把“医保尚未完成”建模为明确、可观察的中间状态，而不是把它误判为入院失败。

## 4. 先持久化可恢复的意图

最小教学模型在一个本地事务中写入：

```text
admission_active_guard：当前在院唯一性
admission：入院登记，insurance_status = PENDING
admission_external_operation：provider、admission_id、status、attempts、last_error
```

提交后可由当前请求同步尝试一次，也可由后台工作器处理医保调用。关键不在“必须异步”，而在于调用前已经保存了可恢复的意图；进程在调用前后崩溃时，仍可扫描 `PENDING` 或可重试失败的操作继续处理。

## 5. 三方幂等与本地恢复缺一不可

医保方必须对外部副作用去重。本地将不可变的 `admissionId` 作为稳定请求身份，例如：

```text
idempotencyKey = HIS:INSURANCE:{orgId}:{admissionId}
```

医保方以同一 key 持久化首次执行结果。重试时返回原结果，而不是再次创建医保登记。

本地仍需保存调用状态，因为超时只表示“本地未收到结果”，不表示医保没有处理。若对方提供按 key 查询结果的接口，优先查询；否则在其幂等承诺成立时，以相同 key 延迟重试。

## 6. 状态决定后续权限

保留本地入院登记不等于忽略医保结果。`INSURANCE_PENDING` 可以允许不依赖医保最终结果的院内流程继续；医保结算、报销资格确认等依赖该事实的动作必须等待成功状态，或按明确的业务规则人工处置。

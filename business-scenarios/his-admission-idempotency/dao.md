# 道：重复传输收敛为同一业务结果

## 道的陈述

当同一业务意图可能因超时、重试或并发被重复传递时，系统应以稳定业务标识绑定已执行结果。因为传输层无法可靠地区分“未执行”和“已执行但响应丢失”，重放同一标识必须返回同一结果而非再次产生副作用。它以持久化状态、查询/重试逻辑和状态机复杂度，交换重复扣费、重复登记或重复库存变更的风险。

## 因果机制

```text
业务意图
-> 不可变业务 ID
-> 原子地记录本地处理权或已执行结果
-> 执行本地或外部副作用
-> 保存可重放结果
-> 同 ID 的重放只读取已有结果或继续恢复
```

本地 guard 只承载“当前在院唯一性”。外部调用记录承载“某协作方处理到哪里、失败原因和重试次数”。两者不能混为一个不断增加状态列的表。

## 适用条件

- 同一业务操作会穿过网络、消息队列或多应用实例。
- 重复到达次数 `N` 可能大于实际独立业务意图数 `K`。
- 副作用不可安全地重复，例如建档、扣款、库存扣减或创建待办。
- 可以定义不可变的业务身份，并让副作用执行者接受该身份。

## 可验证预测

- 对同一 `admissionId` 重试医保调用，不新增医保登记，返回首次结果。
- 两个并发本地登记争抢同一有效在院资格时，数据库只允许一个成功。
- 进程在医保调用前后崩溃后，仍能从 `PENDING` 或失败操作中恢复，不会遗失“需要处理医保”的事实。

## 边界

- `@RepeatedSubmit` 的时间窗口策略不能替代数据库不变量或业务幂等。
- 若医保成功是法律或临床上允许入院的前置条件，则不能先让入院以 `PENDING` 状态继续。
- 同一订单可能对应多次合法支付意图，不能用粗粒度 `orderId` 去重；应为每次独立意图生成 `paymentIntentId`。
- 三方若不提供幂等承诺或查询能力，必须通过对账、人工处置或协商接口契约弥补。

## 迁移映射

| 角色 | 入院医保 | 订单支付 | 消息消费者 |
|---|---|---|---|
| 稳定业务身份 | `admissionId` | `paymentIntentId` | `eventId` |
| 副作用 | 医保登记 | 第三方扣款 | 库存扣减/待办创建 |
| 去重记录 | 医保方首次结果 | 支付平台支付结果 | Inbox/已消费事件表 |
| 可恢复状态 | `INSURANCE_PENDING` | `PAYMENT_PROCESSING` | `PROCESSING` |
| 重放方式 | 查询或同 key 重试 | 查询或同 key 重试 | 队列重复投递 |
| 消费者 | 医保系统 | 支付平台 | 业务消息消费者 |

这不是“选择 UUID”或“增加一个注解”的技巧，而是把不可靠传输造成的重复与不确定性，显式收敛为一个可追踪、可恢复的业务状态。
